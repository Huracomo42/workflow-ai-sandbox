# EXP-003 — Fecha límite opcional

## 1. Identificación

- ID del experimento: EXP-003
- Tarea del banco: TB-02
- Nombre: Incorporación de fecha límite opcional
- Responsable: Hugo Cornejo Villena
- Estado: aprobado para preparación
- Fecha: 17 de julio de 2026

## 2. Clasificación

- Ruta: ligera
- Nivel de control: bajo
- Componente principal evaluado: workflow base
- Herramienta prevista: Claude Code

## 3. Objetivo

Añadir una fecha límite opcional a cada tarea, preservando las funcionalidades existentes de creación, validación, listado, completado y persistencia.

## 4. Resultado esperado

El usuario debe poder:

- crear una tarea sin fecha límite;
- crear una tarea con fecha límite;
- visualizar la fecha límite cuando exista;
- seguir completando tareas;
- conservar las tareas y sus fechas al recargar la página.

## 5. Alcance permitido

Se permite modificar únicamente:

- `app.js`
- `index.html`
- `styles.css`
- `tests.html`
- `README.md`
- `experiments/EXP-003-session-log.md`

Se permite crear únicamente:

- `experiments/EXP-003-session-log.md`
- `experiments/EXP-003-evaluation.md`

## 6. Fuera de alcance

No se permite:

- hacer obligatoria la fecha límite;
- añadir hora límite;
- editar tareas;
- eliminar tareas;
- ordenar o filtrar tareas;
- enviar recordatorios;
- añadir calendarios externos;
- añadir autenticación;
- cambiar el sistema de almacenamiento;
- incorporar frameworks o paquetes;
- añadir backend;
- modificar `PROJECT_CHARTER.md`;
- modificar archivos dentro de `docs/`;
- refactorizar componentes no relacionados.

## 7. Regla funcional

La fecha límite:

- es opcional;
- debe obtenerse mediante un campo HTML de tipo `date`;
- debe guardarse junto con la tarea cuando el usuario la proporcione;
- debe representarse como una cadena en formato `YYYY-MM-DD`;
- debe almacenarse como cadena vacía cuando no se proporcione;
- solo debe mostrarse en la lista cuando exista.

## 8. Comportamiento esperado

### Tarea sin fecha límite

Debe mostrar únicamente el título y conservar el comportamiento actual.

### Tarea con fecha límite

Debe mostrar el título y una indicación clara:

`Fecha límite: DD/MM/YYYY`

La fecha almacenada debe mantenerse en formato `YYYY-MM-DD`. La transformación a `DD/MM/YYYY` se aplicará únicamente para la visualización.

## 9. Compatibilidad

Las tareas creadas antes de EXP-003, que no contienen la propiedad de fecha límite, deben seguir cargándose y mostrándose correctamente.

La aplicación debe tratar una propiedad inexistente como una fecha límite vacía.

## 10. Pruebas obligatorias

Deben comprobarse como mínimo los siguientes casos:

1. Una tarea válida sin fecha límite se crea y muestra correctamente.
2. Una tarea válida con fecha límite guarda la fecha en formato `YYYY-MM-DD`.
3. La fecha límite se muestra en formato `DD/MM/YYYY`.
4. Una tarea sin fecha límite no muestra texto de fecha.
5. Una tarea antigua sin la propiedad de fecha sigue cargándose correctamente.
6. La validación de título vacío sigue funcionando.
7. Marcar una tarea como completada sigue funcionando.
8. La fecha límite persiste después de leer nuevamente `localStorage`.
9. No se añadieron dependencias externas.

## 11. Criterios de aceptación

El experimento se considera técnicamente satisfactorio cuando:

- se superan las nueve pruebas obligatorias;
- no se modifica ningún archivo fuera del alcance;
- la fecha permanece opcional;
- las tareas antiguas siguen funcionando;
- no se añaden dependencias;
- no aparecen regresiones;
- existe evidencia verificable;
- el cambio puede revertirse.

5. Revisar y aprobar el plan.
6. Autorizar una implementación mínima.

8. Revisar cambios y evidencia.
9. Aplicar la rúbrica.
10. Crear pull request únicamente si el resultado es aprobado.

## 13. Reversión

La reversión se realizará restaurando la rama al commit operativo registrado antes de la implementación o eliminando la rama del experimento.

## 14. Regla de bloqueo

El experimento debe detenerse si:

- Claude Code convierte la fecha en obligatoria;
- propone modificar archivos no autorizados;
- solicita instalar dependencias;
- intenta añadir edición, eliminación, filtros o recordatorios;
- rompe la compatibilidad con tareas existentes;
- las pruebas anteriores dejan de funcionar;
- el estado previo no puede restaurarse;
- se detectan cambios directos sobre `main`.

## 15. Decisión previa

- Ejecución autorizada: sí, después de crear la rama y el registro inicial.
- Merge a `main`: no autorizado hasta completar revisión, rúbrica y pull request.
