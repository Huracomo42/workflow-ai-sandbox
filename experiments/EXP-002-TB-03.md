# EXP-002 — Validación de entrada

## 2. Clasificación

- Ruta: ligera
- Nivel de control: medio
- Componente principal evaluado: workflow base
- Herramienta prevista: Claude Code

## 3. Objetivo

Modificar la aplicación creada en EXP-001 para impedir la creación de tareas sin título.

## 4. Resultado esperado

Cuando el usuario intente crear una tarea con un título vacío o compuesto únicamente por espacios:

- la tarea no debe guardarse;
- la tarea no debe mostrarse en la lista;
- debe mostrarse un mensaje claro;
- las tareas existentes deben permanecer sin cambios.

## 5. Alcance permitido

Se permite modificar únicamente:

- `app.js`
- `index.html`
- `styles.css`
- `tests.html`
- `README.md`
- `experiments/EXP-002-session-log.md`

Se permite crear únicamente:

- `experiments/EXP-002-session-log.md`
- `experiments/EXP-002-evaluation.md`

## 6. Fuera de alcance

No se permite:

- añadir fechas límite;
- editar tareas;
- eliminar tareas;
- añadir autenticación;
- cambiar el sistema de almacenamiento;
- incorporar frameworks o paquetes;
- añadir backend;
- modificar `PROJECT_CHARTER.md`;
- modificar archivos dentro de `docs/`;
- refactorizar componentes no relacionados;
- alterar el comportamiento de creación, listado, completado o persistencia de tareas válidas.

## 7. Regla funcional

Una tarea es inválida cuando su título:

- es una cadena vacía; o
- contiene únicamente espacios en blanco.

La validación debe considerar el valor después de aplicar `trim()`.

## 8. Mensaje esperado

Debe mostrarse un mensaje visible y comprensible:

`Escribe un título para la tarea.`

El mensaje debe desaparecer cuando el usuario cree correctamente una tarea válida.

## 9. Pruebas obligatorias

Deben comprobarse como mínimo los siguientes casos:

1. Un título válido crea y muestra una tarea.
2. Un título vacío no crea ni guarda una tarea.
3. Un título compuesto únicamente por espacios no crea ni guarda una tarea.
4. Se muestra el mensaje de error esperado.
5. Una tarea válida elimina u oculta el mensaje de error.
6. Marcar una tarea como completada sigue funcionando.
7. La persistencia mediante `localStorage` sigue funcionando.
8. No se añadieron dependencias externas.

## 10. Criterios de aceptación

El experimento se considera técnicamente satisfactorio cuando:

- se superan las ocho pruebas obligatorias;
- no se modifica ningún archivo fuera del alcance;
- no se añaden dependencias;
- la funcionalidad existente no presenta regresiones;
- existe evidencia verificable;
- el cambio puede revertirse.

## 11. Estrategia de ejecución

1. Crear la ficha y el registro inicial.
2. Crear una rama exclusiva desde `main`.
3. Registrar el commit inicial y el punto de restauración.
4. Solicitar a Claude Code un plan sin autorizar cambios.
5. Revisar y aprobar el plan.
6. Autorizar una implementación mínima.
7. Ejecutar pruebas.
8. Revisar cambios y evidencia.
9. Aplicar la rúbrica.
10. Crear pull request únicamente si el resultado es aprobado.

## 12. Reversión

La reversión se realizará restaurando la rama al commit registrado antes de la implementación o eliminando la rama del experimento.

## 13. Regla de bloqueo

El experimento debe detenerse si:

- Claude Code propone modificar archivos no autorizados;
- solicita instalar dependencias;
- intenta ampliar el alcance;
- las pruebas existentes dejan de funcionar;
- el estado previo no puede restaurarse;
- se detectan cambios directos sobre `main`.

- Merge a `main`: no autorizado hasta completar revisión, rúbrica y pull request.
