# EXP-001 — Creación inicial de una aplicación mínima

## 1. Identificación

- ID del experimento: EXP-001
- Tarea asociada: TB-01
- Nombre: Creación inicial de una aplicación mínima
- Responsable: Hugo Cornejo Villena
- Estado: aprobado
- Fecha de diseño: 17 de julio de 2026

## 2. Propósito

Validar si el workflow base permite crear una aplicación mínima funcional manteniendo planificación, control de alcance, pruebas, documentación y trazabilidad.

## 3. Hipótesis

Si se aplica el workflow definido antes de ejecutar código, será posible crear una aplicación mínima funcional sin cambios fuera de alcance, con pruebas verificables y reversión documentada.

## 4. Clasificación

- Ruta de trabajo: estándar
- Nivel de control: medio
- Componente principal evaluado: workflow base
- Herramienta de ejecución prevista: Claude Code
- Repositorio: `Huracomo42/workflow-ai-sandbox`

## 5. Tarea

Crear una aplicación local y mínima de gestión de tareas que permita:

- crear una tarea;
- listar tareas;
- marcar una tarea como completada.

## 6. Decisión tecnológica

Se utilizará:

- HTML;
- CSS;
- JavaScript puro;
- almacenamiento local del navegador mediante `localStorage`.

No se utilizarán frameworks, backend, base de datos externa ni dependencias adicionales.

Esta decisión reduce complejidad y permite evaluar el workflow sin introducir variables técnicas innecesarias.

## 7. Estado inicial

- Rama base: `main`
- Aplicación existente: ninguna
- Código funcional existente: ninguno
- Dependencias existentes: ninguna
- Pruebas existentes: ninguna
- Documentación existente: Project Charter y paquete operativo

El commit inicial deberá registrarse antes de crear la rama.

## 8. Alcance permitido

Se permite crear:

- `index.html`
- `styles.css`
- `app.js`
- `tests.html`
- actualización del `README.md`
- documentos de evidencia del experimento

Se permite usar únicamente APIs estándar del navegador.

## 9. Alcance no permitido

No se permite:

- añadir frameworks;
- añadir paquetes externos;
- crear backend;
- usar una base de datos externa;
- añadir autenticación;
- desplegar la aplicación;
- modificar el Project Charter;
- modificar instrumentos operativos;
- modificar archivos fuera del alcance aprobado;
- trabajar directamente sobre `main`.

## 10. Requisitos funcionales

### RF-01 Crear tarea

El usuario podrá ingresar un título y crear una tarea.

### RF-02 Listar tareas

Las tareas creadas deberán mostrarse en una lista visible.

### RF-03 Completar tarea

El usuario podrá marcar una tarea como completada.

### RF-04 Persistencia

Las tareas deberán mantenerse después de recargar la página mediante `localStorage`.

## 11. Requisitos no funcionales

- La aplicación deberá ejecutarse abriendo `index.html`.
- No deberá requerir instalación.
- El código deberá ser comprensible.
- La interfaz deberá ser simple y funcional.
- No deberán existir errores visibles en la consola.
- Los datos utilizados serán ficticios.

## 12. Pruebas obligatorias

Deberán verificarse estos casos:

1. crear una tarea válida;
2. mostrar la tarea creada;
3. marcarla como completada;
4. recargar la página y conservar el estado;
5. ejecutar una página local de pruebas;
6. verificar que no se añadieron dependencias;
7. verificar que solo se modificaron archivos autorizados.

## 13. Resultado esperado

Una aplicación mínima funcional que:

- se ejecute localmente;
- permita crear tareas;
- permita listar tareas;
- permita completar tareas;
- conserve las tareas al recargar;
- cuente con pruebas verificables;
- tenga instrucciones de uso;
- respete el alcance.

## 14. Criterios de éxito

El experimento será exitoso cuando:

- se cumplan todos los requisitos funcionales;
- se ejecuten las pruebas;
- no existan cambios fuera de alcance;
- no se añadan dependencias;
- el README quede actualizado;
- exista registro de sesión;
- exista evidencia verificable;
- la reversión esté documentada;
- la rúbrica alcance al menos 38 de 50;
- seguridad, pruebas y calidad técnica tengan mínimo 3;
- no se active una regla de bloqueo.

## 15. Riesgos

| Riesgo | Probabilidad | Impacto | Control |
|---|---|---|---|
| Complejidad innecesaria | Media | Medio | Prohibir frameworks y backend |
| Cambios fuera de alcance | Media | Alto | Lista cerrada de archivos |
| Pruebas insuficientes | Media | Alto | Casos obligatorios definidos |
| Persistencia defectuosa | Media | Medio | Prueba de recarga |
| Declaración falsa de éxito | Baja | Alto | Exigir evidencia real |

## 16. Procedimiento de ejecución

1. Registrar el commit inicial.
2. Crear la rama `experiment/EXP-001-create-minimal-app`.
3. Crear el registro de sesión.
4. Entregar la instrucción completa a Claude Code.
5. Solicitar un plan antes de modificar archivos.
6. Revisar y aprobar el plan.
7. Autorizar la implementación.
8. Ejecutar pruebas.
9. Revisar archivos modificados.
10. Actualizar el README.
11. Aplicar la rúbrica.
12. Documentar la reversión.
13. Crear pull request.
14. Emitir la decisión final.

## 17. Condiciones de detención

Claude Code deberá detenerse si:

- propone una dependencia;
- propone un framework;
- necesita modificar archivos no autorizados;
- detecta una decisión técnica no definida;
- fallan pruebas críticas;
- necesita credenciales;
- no puede mantener el alcance;
- no puede demostrar el resultado.

## 18. Reversión

- Punto de restauración: commit de `main` anterior a la rama
- Método previsto: eliminar la rama o restaurarla al commit inicial
- Reversión física obligatoria: no, salvo fallo grave o regla de bloqueo
- Validación posterior: confirmar que `main` permanece sin cambios y estable

## 19. Evidencias obligatorias

- registro de sesión;

- rama creada;
- plan de Claude Code;
- archivos modificados;
- resultados de pruebas;
- capturas o logs;
- commits;
- pull request;
- rúbrica;
- decisión final.
